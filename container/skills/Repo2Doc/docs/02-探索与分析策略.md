# 02 — 探索与分析策略

> 本文件由 `SKILL.md` 第二/三阶段引用。定义如何深入分析代码库、如何决定读取哪些文件、以及文档的写作规范。

---

## 文件读取优先级

当完整性检查指出文档不完整时，按以下优先级选择要读取的文件：

| 优先级 | 文件类型 | 示例 | 理由 |
|--------|----------|------|------|
| 🔴 最高 | 入口文件 | `main.py`, `index.ts`, `app.py`, `server.go`, `cmd/main.go` | 理解程序启动和整体结构 |
| 🟠 高 | 核心业务逻辑 | `core/*.py`, `services/*.ts`, `handlers/*.go`, `controllers/*.java` | 理解核心功能实现 |
| 🟡 中 | 数据模型 | `models/*.py`, `types/*.ts`, `entities/*.java`, `schema/*.prisma` | 理解数据结构 |
| 🟢 低 | 路由/API 定义 | `routes/*.py`, `router/*.ts`, `api/*.go` | 理解对外接口 |
| 🔵 较低 | 工具函数 | `utils/*.py`, `helpers/*.ts`, `lib/*.go` | 补充实现细节 |
| ⚪ 最低 | 配置/测试 | `config.py`, `test_*.py`, `*.spec.ts` | 仅在其他都完成后 |

### 每次迭代的文件读取规则

1. **最多读取 10 个文件**（控制每次迭代的 token 消耗）
2. **优先读取核心文件**：入口文件 > 业务逻辑 > 数据模型
3. **避免重复读取**：记录已读取的文件列表，不要重复读取
4. **大文件处理**：超过 500 行的文件，优先阅读前 200 行（通常包含导入和主要类/函数定义）
5. **批量读取相关文件**：如果一个模块由多个文件组成，尽量在同一迭代中全部读取

## 代码分析方法

### 方法 1: 文件内容分析

直接读取源码文件，关注以下要素：

```
导入部分     → 依赖关系、使用的框架/库
类定义      → 核心抽象、设计模式
函数签名     → 公共 API、参数类型
注释/文档字符串 → 设计意图、使用说明
配置常量     → 可配置项、默认行为
错误处理     → 边界条件、异常流程
```

### 方法 2: 代码搜索

使用当前运行时提供的搜索工具（如 `grep`、`glob` 等）定位特定模式的代码：

**搜索入口函数**:
- Python: 搜索 `def main` 或 `if __name__`
- Go: 搜索 `func main` 或 `func Main`
- Node.js: 搜索 `"main"` 字段在 package.json 中

**搜索路由定义**:
- Python: 搜索 `@app.route` / `@router` / `@api`
- TypeScript/JS: 搜索 `router.` 调用

**搜索数据模型**:
- Python: 搜索 `class.*Model` / `class.*Schema`
- TypeScript: 搜索 `interface` / `type.*=` 定义

### 方法 3: 依赖分析

通过 import/require 语句追踪模块间的依赖关系：

```
文件 A import 文件 B → A 依赖 B
文件 A 被多个文件 import → A 是核心模块
没有被任何文件 import 的文件 → 可能是入口点或独立脚本
```

## 文档更新要求

每次迭代更新文档时，必须关注以下四个方面：

### 1. 深化技术原理
- 根据实际代码，解释核心功能是**如何实现**的
- 补充关键算法或设计模式的说明
- 解释代码中的重要设计决策（为什么这样做）

### 2. 完善代码映射
- 确保每个功能都能对应到具体文件路径
- 补充关键类/函数的说明
- 添加文件之间的调用关系说明

### 3. 丰富可视化
- 根据实际代码完善/修正 Mermaid 架构图
- 添加或修正模块依赖关系图
- 补充数据流图

### 4. 强化实用性
- 补充"如果要修改X功能，应该去哪个文件看"的指南
- 增加调试技巧和常见问题
- 完善新人上手的阅读顺序

## 模块说明格式

每个核心模块必须按照以下格式记录：

````markdown
### 📁 {模块名称}
**目录**: `path/to/module/`
**职责**: 一句话说明这个模块做什么

**实现原理**:
这个模块通过 XXX 技术/算法来实现 XXX 功能。
核心的设计思路是...

**核心文件**:
| 文件 | 作用 | 关键内容 |
|------|------|----------|
| `xxx.py` | 入口 | `main()` 函数 |
| `yyy.py` | 处理器 | `Handler` 类 |

**调用关系**:
```mermaid
graph LR
    A["入口"] --> B["处理器"]
    B --> C["存储"]
```

**如果要修改...**:
- 修改 XX 功能：去看 `file.py` 的 `function()` 方法
- 增加 YY 功能：在 `module/` 目录下添加新文件
````

## Mermaid 语法规范（必须遵守）

为确保 Mermaid 图表正确渲染，必须遵循以下规范：

1. **强制使用双引号包裹所有节点文本**
   - ✅ `A["服务层(Service)"]`
   - ❌ `A[服务层(Service)]`

2. **节点 ID 保持简单**
   - ✅ `Node1`, `ServiceA`, `DB1`
   - ❌ `node-service-layer`, `数据库`

3. **避免特殊字符冲突**
   - 括号 `()`、方括号 `[]`、花括号 `{}` 必须在双引号内
   - 连接线标签也用双引号：`A -->|"调用"| B`

4. **子图规范**
   - `subgraph "业务层"` — 名称必须用双引号

5. **箭头规范**
   - 实线箭头: `-->`
   - 虚线箭头: `-.->`
   - 粗线箭头: `==>`

6. **换行规范**
   - 节点文本内换行使用 `<br>`

## 写作规范

1. **解释"为什么"**：不只是描述"是什么"，要解释设计决策的原因
2. **通俗语言**：用简单的语言解释复杂概念，避免过度使用术语
3. **相对路径**：所有代码路径使用相对于仓库根目录的相对路径
4. **标注不确定**：不确定的内容标注 `⚠️ 待确认`
5. **Emoji 导航**：使用 emoji 增强可读性（📖 概述、🏗️ 架构、📁 代码、🧩 模块、🚀 上手）
6. **代码片段**：对关键逻辑，提供简化版的伪代码或核心代码片段
7. **中文为主**：文档主体使用中文，技术名词保留英文
