> 本文件由 `SKILL.md` 第一阶段引用。定义如何收集代码库的高层信息，为后续文档生成提供基础素材。

## 步骤 0: 输入类型判断与远程仓库 Clone

用户输入可能是**本地路径**或**远程 Git URL**。先判断类型再处理。

### 判断规则

```
输入内容匹配以下任一模式 → 视为 Git URL:
  - https://github.com/...
  - https://gitlab.com/...
  - git@github.com:...
  - 任何以 .git 结尾的 URL
  - 任何以 https:// 或 git:// 开头且包含仓库路径的 URL

否则 → 视为本地路径，跳到步骤 1
```

### Git 环境检查

如果判定为 Git URL，先验证 git 可用：

```bash
git --version
```

如果命令失败，**立即停止**并提示用户：
```
❌ git 未安装或不可用。
要分析远程仓库，需要先安装 git。
安装后重新运行，或者手动 clone 后提供本地路径。
```

### Clone 流程

```bash
# 1. 确定 clone 目标目录
#    默认: ~/repo2doc-repos/{repo_name}
#    repo_name 从 URL 提取，如 https://github.com/user/my-project.git → my-project

# 2. 检查目标目录是否已存在
#    如果已存在 → 询问用户：使用现有目录 / 重新 clone（删除后重新拉取）/ 取消
#    如果不存在 → 直接 clone

# 3. 执行 clone（浅克隆，节省时间和空间）
git clone --depth 1 {git_url} ~/repo2doc-repos/{repo_name}

# 4. 确认 clone 成功
#    成功 → repo_path = ~/repo2doc-repos/{repo_name}
#    失败 → 提示用户检查 URL、网络连接、仓库权限
```

**Clone 参数说明**：
- `--depth 1`：浅克隆，只拉取最新一次提交，大幅减少下载量。对于代码分析足够用。
- 如果用户需要完整历史（如分析 git log），可去掉 `--depth 1`。

**私有仓库处理**：
- 如果 clone 失败且错误信息包含 `Authentication failed` 或 `Permission denied`：
  ```
  ⚠️ 该仓库可能是私有仓库，需要认证。
  请选择：
  1. 使用 HTTPS + token: git clone https://{token}@github.com/user/repo.git
  2. 使用 SSH: git clone git@github.com:user/repo.git（需要配置 SSH key）
  3. 手动 clone 后提供本地路径
  ```

### 示例

```
用户: "帮我分析 https://github.com/fastapi/fastapi"

→ 检测到 Git URL
→ git --version ✅
→ repo_name = fastapi
→ git clone --depth 1 https://github.com/fastapi/fastapi ~/repo2doc-repos/fastapi
→ repo_path = ~/repo2doc-repos/fastapi
→ 继续步骤 1
```

## 步骤 2: 读取 README

按优先级查找并读取以下文件（取第一个找到的）：

```
README.md → readme.md → README.rst → README.txt → README
```

**处理规则**：
- 如果 README 超过 10000 字符，截取前 10000 字符并标注 `... (内容已截断)`
- 若未找到任何 README 文件，记录 `（未找到 README 文件）`
- 提取 README 前 3000 字符作为高层信息汇总使用

## 步骤 3: 生成目录结构树

使用终端命令获取目录结构，根据操作系统选择命令：

**Windows**:
```bash
tree /F /A "{repo_path}"
```

**Linux/Mac**:
```bash
find "{repo_path}" -maxdepth 3 \
  -not -path '*/\.*' \
  -not -path '*/node_modules/*' \
  -not -path '*/__pycache__/*' \
  -not -path '*/.venv/*' \
  -not -path '*/venv/*' \
  | head -200
```

### 排除目录列表

以下目录应被忽略，不纳入分析：

```
node_modules     # JS 依赖
.git             # Git 版本控制
venv / .venv     # Python 虚拟环境
__pycache__      # Python 缓存
dist / build     # 构建产物
vendor           # 第三方依赖
.cache           # 缓存
.idea / .vscode  # IDE 配置
target           # Java/Rust 构建目录
bin / obj        # .NET 构建目录
.next / .nuxt    # 框架构建缓存
coverage         # 测试覆盖率
.pytest_cache    # pytest 缓存
.tox             # tox 测试
```

## 步骤 4: 收集项目配置文件

检查以下配置文件是否存在，读取存在的文件内容（每个文件限制 3000 字符）：

### 按语言/框架分类

| 类型 | 配置文件 | 重要信息 |
|------|----------|----------|
| **Python** | `pyproject.toml`, `setup.py`, `setup.cfg`, `requirements.txt`, `Pipfile` | 依赖列表、项目元数据 |
| **JavaScript/TS** | `package.json`, `tsconfig.json` | 依赖、脚本命令 |
| **Java** | `pom.xml`, `build.gradle`, `build.gradle.kts` | 依赖、项目结构 |
| **Rust** | `Cargo.toml` | 依赖、workspace 结构 |
| **Go** | `go.mod`, `go.sum` | 模块路径、依赖 |
| **Ruby** | `Gemfile` | 依赖 |
| **PHP** | `composer.json` | 依赖 |
| **通用** | `Makefile`, `CMakeLists.txt` | 构建命令 |
| **容器** | `Dockerfile`, `docker-compose.yml` | 部署架构 |
| **环境** | `.env.example`, `.env.sample` | 配置项清单 |

### 优先读取的文件

以下文件应**完整读取**内容（限 3000 字符），因为它们包含重要的项目元数据：

```
pyproject.toml, package.json, pom.xml, build.gradle,
Cargo.toml, go.mod, composer.json
```

其他文件只需确认存在即可。
